# Project AppMigration: Cloud-Native Deployment Auditing Platform
[![Build and Push to Amazon ECR](https://github.com/Hero-Xero/AppMigration/actions/workflows/ci.yml/badge.svg)](https://github.com/Hero-Xero/AppMigration/actions/workflows/ci.yml)

Project AppMigration is the complete modernization of a legacy, on-premises metrics and auditing tool. The original application was used to scrape and track legacy internal deployment activity. This project represents its migration to a fully cloud-native architecture on Amazon Elastic Kubernetes Service (EKS), completely rebuilt to audit and track modern, cloud-native deployments and pipeline outcomes.

The resulting application is a Spring Boot service that acts as an immutable ledger for DevOps activity. The platform is deployed using a GitOps delivery model, with the full infrastructure stack, from networking to observability and alerting, provisioned as code through Terraform.

## What the Application Does

At its core, the application operates purely as a backend service with no graphical interface, focusing entirely on data ingestion and observability. Its primary responsibilities are:

* **Deployment Auditing:** Records every deployment event for tracked applications, including image tags, timestamps, and the outcomes of the associated CI/CD pipeline runs.
* **State Persistence:** Stores the complete deployment history in Amazon RDS (PostgreSQL), providing a queryable audit trail for post-incident analysis and compliance.
* **Self-Observability:** Exposes rich Micrometer/Prometheus-compatible metrics (JVM memory, garbage collection pauses, HTTP request latency) to Grafana, ensuring the auditing system itself remains highly observable.
* **Self-Auditing CI/CD:** The application's own GitHub Actions pipeline calls its API after every successful build, so each real deployment of AppMigration is itself recorded in the audit ledger it maintains.

In short: while other systems are responsible for shipping code, this application provides the persistent, verifiable record of exactly what happened, when it happened, and how the infrastructure responded.

## System Architecture

Infrastructure is provisioned entirely through Terraform and follows a GitOps model in which this repository is the source of truth for both application manifests and cluster state.

| Category | Technology | Purpose |
| --- | --- | --- |
| Cloud Provider | Amazon Web Services (AWS) | Target infrastructure environment |
| Infrastructure as Code | Terraform | Provisioning of VPC, EKS, RDS, IAM, and supporting AWS resources |
| Backend API | Spring Boot 3.x (Java 21) | Core auditing logic and metrics processing |
| Database | Amazon RDS (PostgreSQL) | Persistent storage for deployment and pipeline metrics |
| Container Registry | Amazon ECR | Docker image hosting for the application |
| CI/CD Pipeline | GitHub Actions | Automated testing, image builds, manifest updates, and audit-trail recording |
| Container Orchestration | Amazon EKS | Managed Kubernetes cluster |
| GitOps Controller | ArgoCD | Pull-based, automated cluster synchronization |
| Ingress & Load Balancing | AWS Load Balancer Controller (ALB) | Provisions and manages internet-facing Application Load Balancers for the app, ArgoCD, and Grafana |
| Observability (Metrics) | Prometheus & Grafana | Time-series metrics collection and dashboarding via kube-prometheus-stack |
| Alerting | Prometheus Alertmanager & Discord | Routes critical alerts to a Discord channel via a native webhook receiver |
| Log Aggregation | Fluent Bit & CloudWatch | Centralized pod log management via IAM Roles for Service Accounts (IRSA) |
| Secrets | Kubernetes Secrets | Database and Grafana credentials, generated by Terraform and injected at provision time |

## Core Database Entities

The application tracks the software lifecycle through the following entities:

| Entity | Attributes | Relationships |
| --- | --- | --- |
| Application | id (UUID), name, description, gitRepoUrl, createdAt, updatedAt | One-to-Many with Deployment |
| Deployment | id (UUID), timestamp, imageTag, status | Many-to-One with Application, One-to-One with Pipeline Run, One-to-Many with Monitoring Metric |
| Pipeline Run | id (UUID), commitSha, branch, triggeredBy, outcome | One-to-One with Deployment |
| Monitoring Metric | id (UUID), metricName, value, recordedAt | Many-to-One with Deployment |

## Audit Ingestion API

The auditing trail is populated through a small set of REST endpoints, called automatically by the CI/CD pipeline on every successful build:

* `POST /api/applications` — Creates an Application record. This endpoint is idempotent: it looks up an existing Application by name first and returns it if found, rather than creating a duplicate, which allows the CI pipeline to call it safely on every run without producing repeated records for the same application.
* `POST /api/applications/{id}/deployments` — Records a new Deployment under a given Application, including the image tag pushed to ECR and its status.
* `POST /api/deployments/{id}/pipeline-runs` — Attaches the outcome of the CI/CD pipeline run (commit SHA, branch, trigger source, and outcome) to a specific Deployment.
* `POST /api/deployments/{id}/metrics` — Records point-in-time monitoring metrics against a specific Deployment.

Read endpoints are provided alongside each of the above for retrieving Applications, Deployments, Pipeline Runs, and Metrics by ID.

## Networking and Ingress

All internet-facing components (the application API, ArgoCD, and Grafana) are exposed through separate Application Load Balancers, each provisioned by the AWS Load Balancer Controller from a dedicated Kubernetes Ingress resource. There is no single shared load balancer; each component receives its own ALB and its own DNS hostname, generated dynamically at provision time.

For the AWS Load Balancer Controller to discover and manage these resources correctly, the VPC's public subnets are tagged with:

* `kubernetes.io/role/elb = 1`
* `kubernetes.io/cluster/<cluster-name> = shared`

The controller's IAM role (attached via IRSA) is granted the permissions defined in the official AWS Load Balancer Controller IAM policy, supplemented with a small additional policy for actions not yet covered by that baseline (`ec2:DescribeRouteTables`, `elasticloadbalancing:DescribeListenerAttributes`, `elasticloadbalancing:DescribeCapacityReservation`). Health checks for Grafana and ArgoCD are configured explicitly via Ingress annotations, since both applications redirect unauthenticated requests to a login route rather than returning a plain `200 OK` at the root path.

ArgoCD is served over plain HTTP rather than HTTPS, since no custom domain or TLS certificate is provisioned for this environment. This required disabling the chart's built-in Ingress template, which hardcodes an unusable placeholder hostname, in favor of a hand-written Ingress resource targeting ArgoCD's insecure HTTP port directly.

## GitOps Deployment Strategy

The application itself is deployed using a pull-based, zero-touch GitOps workflow:

1. **Continuous Integration:** Code pushed to the main branch triggers a GitHub Actions workflow that runs tests, builds and scans the Docker image, and pushes it to Amazon ECR.
2. **Manifest Update:** The pipeline updates the image tag in the Kubernetes deployment manifest and commits the change back to the repository.
3. **Continuous Delivery:** ArgoCD, running inside the EKS cluster, continuously compares the live cluster state against the manifests in this repository.
4. **Reconciliation:** On detecting drift, ArgoCD automatically applies the updated manifests, triggering a rolling update of the deployment. The ArgoCD Application is configured with automated sync and self-healing to revert any manual cluster changes back to the state defined in Git.
5. **Audit Recording:** After the manifest update succeeds, the pipeline calls the application's own API to record the Application, Deployment, and Pipeline Run for this build, closing the loop between "code was shipped" and "the audit ledger reflects it."

## Observability

Prometheus and Grafana are deployed into a dedicated `monitoring` namespace via the `kube-prometheus-stack` Helm chart. Grafana is reachable through its own ALB-backed Ingress, with credentials generated at provision time and exposed as sensitive Terraform outputs.

Grafana is additionally configured with a direct PostgreSQL data source pointed at the application's RDS instance, provisioned automatically through Terraform at install time using the same generated database credentials the application itself uses. This allows the Grafana dashboard to query the audit ledger directly, alongside the application's JVM and HTTP metrics, with no manual data source configuration required.

The default dashboard includes:

* A deployment ledger table, querying Application, Deployment, and Pipeline Run records directly from PostgreSQL
* Process CPU usage
* Garbage collection pause time
* Heap and non-heap memory usage
* HTTP request rate by endpoint

Fluent Bit runs as a DaemonSet across all nodes and forwards container logs to CloudWatch Logs, using an IAM role scoped specifically to log write permissions via IRSA to avoid long-lived credentials on the nodes.

## Alerting

Prometheus Alertmanager is configured with a Discord receiver, so critical alerts are delivered directly to a Discord channel without any intermediate service. The default routing tree sends all alerts to Discord by default, with the built-in `Watchdog` alert (a synthetic, always-firing check used to confirm the alerting pipeline itself is functioning) explicitly routed to a no-op receiver instead, matching the chart's default behavior.

A custom alerting rule, `AppMigrationDown`, is defined alongside the default rule set and fires if the application's Prometheus target becomes unreachable, providing an immediate notification if the service goes down.

## Security and Reliability

* **Principle of Least Privilege:** The RDS security group accepts PostgreSQL connections (TCP 5432) strictly from the EKS node security group. It is not reachable from the public internet.
* **Scoped IAM via IRSA:** The AWS Load Balancer Controller and Fluent Bit each run under their own IAM role, bound to a specific Kubernetes service account through OIDC federation.
* **Concurrency Protection (Database Initialization):** The application currently runs a single replica to safely manage database schema initialization via Hibernate. This is a deliberate limitation to prevent race conditions during startup, with plans to decouple schema management using a dedicated migration tool (e.g., Flyway) to enable safe multi-replica scaling.
* **OOM Prevention:** JVM heap limits are set explicitly (`-Xmx256m -Xms128m`) and paired with matching Kubernetes resource requests and limits to prevent pod eviction under memory pressure.
* **Secrets Handling:** Database and Grafana credentials are generated dynamically by Terraform and injected into the cluster as Kubernetes Secrets, and into Grafana's data source configuration via secure JSON fields rather than plaintext.

## Infrastructure Lifecycle (Teardown Process)

Because several components (the AWS Load Balancer Controller, ArgoCD, and Grafana) provision AWS resources, such as load balancers, target groups, and security groups, outside of Terraform's own state, tearing down the environment requires a specific sequence. AWS will refuse to delete a VPC if any external elastic network interface or non-default security group still references it.

A dedicated teardown script handles this by:

1. Deleting Ingress and LoadBalancer-type Service objects while the EKS cluster is fully operational, signaling the controller to cleanly delete the physical AWS load balancers.
2. Sweeping the target VPC directly to force-delete any orphaned ENIs and resolve cyclic security-group-to-security-group references that the controller may leave behind.
3. Running the Terraform destruction phase with automatic state-lock recovery, retrying the sweep if a dependency violation is encountered, to ensure the VPC is completely stripped of dependencies before Terraform attempts to delete it.

## Local Development and Provisioning

**Prerequisites**

* AWS CLI, installed and authenticated
* Terraform >= 1.5
* kubectl
* Helm
* Docker

**Provisioning the Infrastructure**

```bash
cd terraform
terraform init
terraform apply
```

On completion, Terraform outputs the application's ECR repository URL, the RDS endpoint, and the public URLs and credentials for the application, Grafana, and ArgoCD.

**Tearing Down the Infrastructure**

```bash
./teardown.sh
```

*This script must be used instead of a direct `terraform destroy` to prevent AWS dependency violations and VPC deadlocks.*
